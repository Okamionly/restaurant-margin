## Bug du jour

**Fichier** : `client/src/types/index.ts` + `client/src/components/NPSModal.tsx`  
**Sévérité** : Moyenne (type safety, silencing d'erreur potentielle)

### Problème

L'endpoint `/api/auth/me` (`api-lib/routes/auth.ts:337`) retourne le champ `createdAt` dans la réponse utilisateur. Cependant, l'interface `User` dans `client/src/types/index.ts` ne déclarait pas ce champ.

Conséquence : `NPSModal.tsx` était forcé d'utiliser deux casts `(user as any).createdAt` pour accéder à la date d'inscription (logique NPS J+14), contournant le typage TypeScript et risquant de silencer de futures erreurs.

### Avant

```typescript
// types/index.ts
export interface User {
  id: number;
  email: string;
  name: string;
  role: string;
  plan?: string;
  emailVerified?: boolean;
  trialEndsAt?: string | null;
  // createdAt absent
}

// NPSModal.tsx
if (!user?.id || !(user as any)?.createdAt) return;
const signupDate = new Date((user as any).createdAt);
```

### Après

```typescript
// types/index.ts
export interface User {
  id: number;
  email: string;
  name: string;
  role: string;
  plan?: string;
  emailVerified?: boolean;
  trialEndsAt?: string | null;
  createdAt?: string;  // ← ajouté
}

// NPSModal.tsx
if (!user?.id || !user.createdAt) return;
const signupDate = new Date(user.createdAt);
```

### Raison du choix

- Fix simple (2 fichiers, 3 lignes)
- Alignement type/API réel — le champ existe côté backend
- Suppression des `as any` améliore la sécurité du typage
- L'endpoint debug Stripe (P0 CTO) avait déjà été supprimé avant ce run
