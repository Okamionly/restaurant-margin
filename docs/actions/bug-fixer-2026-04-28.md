# Bug Fixer — 2026-04-28

## Bug fixé : Fuite d'informations internes dans la réponse d'erreur Stripe

### Fichier
`api/index.ts` — ligne ~354

### Contexte
Le CTO a identifié un endpoint debug Stripe exposé en production (`/api/debug/stripe-fetch`, commit `44ebaab`).
Cet endpoint avait déjà été supprimé dans un commit antérieur.

En revanche, le gestionnaire d'erreur du endpoint `/api/stripe/checkout` retournait encore des détails internes dans le corps JSON de la réponse 500 :
- `details: err.message` — message d'erreur brut
- `debug.code`, `debug.type`, `debug.statusCode`, `debug.causeMessage`, `debug.causeCode`, `debug.causeErrno`

Ces informations fuient vers le navigateur client en cas d'erreur Stripe, exposant la structure interne et potentiellement des détails réseau/API.

### Avant
```json
{
  "error": "Erreur création session Stripe",
  "details": "<message d'erreur interne>",
  "debug": {
    "code": "...",
    "type": "...",
    "statusCode": "...",
    "causeMessage": "...",
    "causeCode": "...",
    "causeErrno": "..."
  }
}
```

### Après
```json
{
  "error": "Erreur création session Stripe"
}
```

Les `console.error` côté serveur sont conservés (logs Vercel) pour le diagnostic.

### Raison
Sécurité — principe du minimum d'information exposée. Une erreur 500 ne doit retourner que le message générique, jamais les détails internes.
